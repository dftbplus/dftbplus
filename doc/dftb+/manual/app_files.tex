\chapter{Description of some relevant file formats}
\label{app:files}

\section{The XYZ File Format}

The \emph{XYZ file format} is a minimalist, plain-text format. It was
never formalised by a standards organisation. Instead, evolving
organically, with minor variations appearing in different codes,
different implementations of incompatible extensions, particularly
with respect to the number and interpretation of columns in the atomic
data section.

\subsection{Canonical Structure}

An XYZ file describing a system of $N$ atoms has the following
structure:

\begin{enumerate}
  \item The first line contains the integer number of atoms $N$.
  \item The second line is a comment line containing arbitrary text.
  \item The following $N$ lines contain atomic data.
\end{enumerate}

The comment line is frequently left empty but is syntactically
required by most readers. Some implementations overload it with
metadata such as energies or timestep data.

\subsection{Four-Column XYZ Format}

The most common and widely supported variant is the four-column XYZ
format.  Each atomic line consists of the atomic symbol followed by
three Cartesian coordinates, typically in \AA{}:
\[
\text{symbol}_i \quad x_i \quad y_i \quad z_i.
\]

A typical example is
\begin{verbatim}
3
Water molecule
O   0.000000   0.000000   0.000000
H   0.758602   0.000000   0.504284
H  -0.758602   0.000000   0.504284
\end{verbatim}

Here, atomic identity is specified by the chemical symbol
(case-sensitive by convention).

\subsection{Five-Column XYZ Format}

A five-column XYZ file extends the atomic line with an additional
per-atom quantity:
\[
\text{symbol}_i \quad x_i \quad y_i \quad z_i \quad q_i.
\]

The fifth column is most commonly interpreted as a partial atomic
charge. Because the meaning of the fifth column is not standardised,
files using this extension are not reliably portable across programs.

\subsection{Seven-Column XYZ Format}

The seven-column XYZ format is most frequently encountered in
molecular dynamics contexts. In this variant, three additional columns
are appended to store a Cartesian vector quantity for each atom:
\[
\text{symbol}_i \quad x_i \quad y_i \quad z_i \quad v_{x,i} \quad
v_{y,i} \quad v_{z,i}.
\]

These quantities are most often atomic velocities, though forces or
other vectors can also be used in practice.

A representative example is
\begin{verbatim}
2
Diatomic molecule with velocities
H   0.000000   0.000000   0.000000   0.010   0.000   0.000
H   0.740000   0.000000   0.000000  -0.010   0.000   0.000
\end{verbatim}

As with the five-column case, interpretation of the extra columns
depends entirely on external conventions.

\section{XYZ Formats in DFTB+}

\subsection{Introduction}

DFTB+ supports the XYZ family of formats primarily as a \emph{geometry
exchange and output format}, rather than as its most feature-complete
internal input representation. While the DFTB+ native geometry format
(\texttt{.gen}, see \ref{app:gen}) provides explicit support for atom
types, lattice vectors, and periodicity, XYZ files are supported for
both input and output in several distinct variants.

These variants differ in the number of columns and in the
interpretation of the comment line, depending on the mode of operation
(single-point calculation, geometry optimisation, or molecular
dynamics).

\subsection{XYZ as an Input Geometry}

\subsubsection{Four-Column XYZ (Input Mode)}

DFTB+ accepts standard four-column XYZ files as geometry input when
the \texttt{Geometry = XYZ \{ ... \}} block is used. In this mode,
each atomic line has the form
\[
\text{symbol}_i \quad x_i \quad y_i \quad z_i,
\]
with Cartesian coordinates given in \AA{} for the $i = 1 \ldots N$
lines.

The atomic symbol is mapped internally to a DFTB+ \emph{species} via
the \iscb{Species} definition block in the \verb|dftb_in.hsd| input
file. No charge, velocity, or constraint information can be read from
an XYZ input.

Periodic boundary conditions cannot be specified in pure XYZ input;
systems read in this way are treated as non-periodic unless lattice
information is provided elsewhere in the input.

\subsection{XYZ Output}

\subsubsection{XYZ Output: Geometry Optimisation}

During geometry optimisation, \dftbp{} writes intermediate structures
using the XYZ format. Each optimisation step is (optionally) written
as a separate XYZ ``frame'' into the file.  This produces a
concatenated XYZ trajectory, compatible with common molecular
visualisation tools.

For non-self-consistent calculations, this file uses the conventional
four-column format, while for \kw{SCC}, the five column format with
Mulliken population (not charge) is produced. For \kw{SCC} or if
Mulliken populations are enabled, the five column format with Mulliken
population (not charge) is produced.

Note: In the special case of optimisation with non-collinear or
spin-orbit coupling enabled, the atomic spins (in units of electron
charges) are output in the last 3 columns.

\subsubsection{XYZ Output: Molecular Dynamics}

In molecular dynamics simulations, DFTB+ can write extended XYZ
trajectory files that include the atomic velocities in the last 3
columns.  Coordinates are given in \AA{}, and velocities are given in
\AA{}/ps. Frames are written every \kw{MDRestartFrequency} steps in
the MD.

Again, if Mulliken or SCC are enabled, the populations are given as
the 4$^\mathrm{th}$ column of the file.
